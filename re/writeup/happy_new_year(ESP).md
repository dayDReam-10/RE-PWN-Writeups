先说一下upx压缩壳原理，其实是用了pushad把寄存器先保存到栈顶，然后执行加密压缩什么的操作

由于程序最后一定能执行，所以我们知道这个upx一定存在还原点，我们的任务就是找出还原点（popad）

```text
【内存地址】       【栈内存里的内容】
0019FF70          [ 原始程序的数据 A ]  <-- 初始 ESP (原来的数据)
---------------------------------------
0019FF6C          [ EAX 备份 ]
0019FF68          [ ECX 备份 ]
...               ...
0019FF50          [ EDI 备份 ]         <-- 壳运行时的 ESP (我们的断点就在这)
=======================================
           ↓↓↓ 执行 POPAD 瞬间 ↓↓↓
=======================================
1. CPU 从 0019FF50 开始，把 8 个备份依次“吸”回寄存器。
2. 弹簧（ESP）瞬间向上弹回 32 字节 (8个寄存器 * 4字节)。
3. ESP 重新指向了 0019FF70 
```

然后我们要做的就是保存好这些还原后的数据（dump）

## 以下是步骤

我们先要找壳的入口

按两次f9（RUN）到达 `0040E2F0 AddressofEntrypoint` 

我们要看pushad过后栈顶在哪

**Q：为什么不能看pushad之前的栈顶就好了？**

**A：**
当一个加壳程序（比如 UPX）刚加载进内存时，它还没开始解压。此时的 ESP 指向的是 **操作系统（OS）** 为这个进程分配的初始栈顶。

这个栈顶里的数据是 **Windows 系统调用** 留下的环境信息、参数等。

对我们要脱的那个“核心程序”来说，这些是系统的东西，不是它自己的东西

那加不加壳有什么区别？

**1没加壳的程序：直接进家门**
当你双击一个没加壳的 `.exe` 时：

* 操作系统把程序加载进内存。
* 直接跳转：操作系统直接把 CPU 的指针（EIP）指到了程序的 真正的入口点 (OEP)。
* 状态：此时你一打开调试器，看到的直接就是 `PUSH EBP / MOV EBP, ESP`。
* 堆栈：确实是操作系统的“垃圾数据”。
* 代码：直接就是原始代码，没有解压过程。

而加壳之后的最后那几行代码是弹出寄存器过后有一个大跳转然后大跳转之后在移动CPU指针EIP

所以！

我们先f8找到加完寄存器的esp

**为什么要这样？**

找到这个新的 ESP 地址后，我们在这个地址上点右键，设置硬件访问断点。
这一步的意义是：给这个内存坑位装一个“红外探测器”

* **现在**：我们已经跳过了“写入”数据的动作。
* **中间过程**：壳程序开始疯狂解压代码，它会用掉成千上万行指令，堆栈也会被它用得乱七八糟。但是，壳程序有个铁律：它绝不敢动 ESP 指向的这块“老巢”区域，因为它还要靠里面的数据复活。
* **结果**：你可以放心按 F9（运行）。程序会像断了线的风筝一样飞快运行，直到它解压完毕。

## 接下来就是 dump

双击跳转，点击scylla

在 Scylla 的底层逻辑里，它是这么干活的：

* **Dump (第一步)**：它是把内存里现有的数据直接镜像到硬盘。这个动作只管“复制粘贴”，它不管代码能不能跑。
* **Fix Dump (第二步)**：它是把**搜到的 IAT（函数表）**重新写进那个镜像好的文件头里

所以先直接dump一次，然后IAT修一下导入表，fix it
